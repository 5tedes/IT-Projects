1. Executive Summary

This document provides a comprehensive plan for migrating Aigaia School of Art & Design's IT services from the current provider (SKWebline Ltd) to a new provider. The migration is targeted for completion by the end of September 2026.

The plan addresses significant concerns about the current provider's reliability, including unresolved configuration errors and a DNS change made without client notification that caused service outages. Given these concerns, this plan includes extensive safeguards against potential vindictive actions or negligence during the transition period.

1.1 Current Situation Overview

The client currently relies on SKWebline Ltd for domain management, DNS, website hosting, and e-commerce services. While Office 365 and the newsletter system (Brevo) are securely under client control, critical infrastructure including DNS records, website files, and the custom e-shop remain accessible only to the provider.

1.2 Key Risks Identified

  • No client access to Cloudflare DNS (controls email delivery and website routing)
  • Two domains (theatroaigaia.com, artshopaigaia.com) with unclear ownership
  • No backups of website files, databases, or e-shop source code
  • Custom e-shop with proprietary code that may be difficult to replicate
  • History of provider making changes without notification

4. Threat Scenarios and Mitigations

This section identifies potential harmful actions the current provider could take (whether through negligence or intent) and provides specific countermeasures.

4.1 Domain-Related Threats

Threat: Allowing domain registration to lapse

The provider could fail to renew theatroaigaia.com or artshopaigaia.com, causing them to expire and potentially be acquired by third parties.

Mitigation:
  • Demand written confirmation of domain registrant details from SKWebline immediately
  • Set calendar reminders 90 days before each expiration date
  • Request transfer of domains to client-controlled registrar before announcing migration
  • If registered to client, contact NETIM directly to verify and potentially gain direct access

Threat: Refusing to unlock domains for transfer

All three .com domains currently have "client transfer prohibited" status. The provider could refuse to remove this lock.

Mitigation:
  • Obtain written agreement for domain transfer before announcing migration intent
  • For NETIM domains: Contact NETIM directly if registered in client's name
  • File ICANN complaint if provider refuses legitimate transfer request
  • Engage legal counsel if significant business impact occurs

Threat: Claiming ownership of domains

If domains are registered in SKWebline's name, they could claim ownership.

Mitigation:
  • Gather all invoices showing client paid for domain registrations
  • Locate original service contract specifying asset ownership
  • Document email correspondence about domain purchases
  • Prepare for potential legal dispute or negotiated buyout

4.2 DNS-Related Threats

Threat: Deleting or modifying DNS records

The provider could delete critical DNS records, instantly taking all services offline (website, email, e-shop).

Mitigation:
  • Document all current DNS records immediately (before announcing migration)
  • Use online tools to query and record current DNS configuration
  • Set up monitoring to alert on DNS changes (e.g., DNS monitoring services)
  • For aigaia.com.cy: Change nameservers to client-controlled DNS before migration announcement

Threat: Pointing domains to malicious or inappropriate content

The provider could redirect the website to harmful, embarrassing, or competitor content.

Mitigation:
  • Transfer DNS control before announcing migration
  • Set up website monitoring to alert on content changes
  • Document any malicious redirects as evidence for potential legal action

4.3 Website and Hosting Threats

Threat: Refusing to provide website files and database backups

The provider could refuse to hand over WordPress files, themes, plugins, and database exports.

Mitigation:
  • Install WordPress backup plugin immediately (e.g., UpdraftPlus, All-in-One WP Migration)
  • Run regular automated backups to client-controlled cloud storage
  • Test backup restoration on a staging environment
  • Use web scraping tools as last resort to capture content

Threat: Deleting website content or defacing the site

The provider could delete or modify website content maliciously.

Mitigation:
  • Create complete backup before any migration communication
  • Archive website using Wayback Machine (archive.org) for historical record
  • Set up visual monitoring to detect changes (e.g., Visualping)

Threat: Shutting down hosting immediately upon notice

The provider could terminate hosting access before migration is complete.

Mitigation:
  • Have new hosting environment fully prepared and tested before announcing migration
  • Ensure all backups are current and tested for restoration
  • Review contract for notice period requirements
  • Document any breach of contract terms for potential legal recourse

4.4 E-Shop Specific Threats (CRITICAL)

⚠️ Highest Risk Area: The custom-built e-shop represents the most significant vulnerability in this migration.

Threat: Refusing to provide e-shop source code

As a custom-built platform, the e-shop code may be considered SKWebline's intellectual property unless the contract specifies otherwise.

Mitigation:
  • Review contract for intellectual property clauses regarding custom development
  • Request source code handover in writing as part of normal service transition
  • If refused, prepare contingency to rebuild on standard e-commerce platform (see Section 7)
  • Document current functionality thoroughly (screenshots, user flows, features list)

Threat: Refusing to provide customer and order database

Loss of customer data and order history would be catastrophic for business continuity.

Mitigation:
  • Export all accessible data through admin panel immediately (products, orders, customers)
  • Request database exports as part of regular "backup" rather than migration
  • Customer data belongs to the client under GDPR - use this as leverage if necessary
  • Maintain separate customer list export updated regularly

Threat: Breaking e-shop functionality or payment processing

The provider could disable checkout, break payment integration, or cause data corruption.

Mitigation:
  • Monitor e-shop functionality daily during transition period
  • Have alternative sales channel ready (even manual order taking) as fallback
  • Verify payment gateway credentials are under client control

4.5 Email-Related Threats

Threat: Breaking email delivery (MX/SPF/DKIM records)

The provider could modify DNS records to break email delivery. Note: DKIM is already broken since the January 2026 DNS change.

Mitigation:
  • Document all current email-related DNS records (MX, SPF, DKIM, DMARC)
  • Office 365 is client-controlled, so email service itself is safe
  • Priority action: Fix DKIM records and transfer DNS control to restore full email functionality
  • Once DNS is client-controlled, provider cannot affect email delivery

4.6 Data and Privacy Threats

Threat: Retaining or misusing customer data

The provider may retain copies of customer data after termination.

Mitigation:
  • Include data deletion requirement in termination notice
  • Request written confirmation of data deletion under GDPR
  • Review contract for data processing agreement terms

Threat: Accessing accounts after termination

The provider may attempt to retain access to services.

Mitigation:
  • Change all passwords immediately after migration completion
  • Review and revoke all API keys and access tokens
  • Remove SKWebline from Office 365 partner relationships
  • Audit all service access logs for 90 days post-migration

5. Immediate Actions (Before Announcing Migration)

⚠️ Important: These actions should be completed BEFORE informing SKWebline of the intention to migrate. This is critical to secure assets while cooperation is still assumed.

5.1 Week 1-2: Documentation and Evidence Gathering

  1. Locate and secure all contracts and agreements with SKWebline
  2. Gather all invoices related to domain registrations and hosting
  3. Document all email correspondence regarding services
  4. Screenshot all admin panel access (WordPress, e-shop, Office 365)
  5. Record all current DNS records for all domains using online lookup tools
  6. Archive all websites using Wayback Machine

5.2 Week 2-4: Securing Backups

  1. Install WordPress backup plugin and run full backup to external storage
  2. Export all e-shop data accessible through admin (products, categories, orders, customers)
  3. Document all e-shop functionality with screenshots and user flow diagrams
  4. Export all Brevo subscriber lists and campaign history
  5. Export Office 365 data (mailboxes, SharePoint, OneDrive) as additional safety measure
  6. Test WordPress backup restoration on a separate hosting account

5.3 Week 4-6: Domain Ownership Clarification

  1. Contact NETIM directly to verify registrant details for theatroaigaia.com and artshopaigaia.com
  2. If registered to client: Request direct account access from NETIM
  3. If registered to SKWebline: Prepare documentation proving client paid for registrations
  4. Verify aigaia.com.cy renewal is on calendar (expires 31/12/2026)
  5. Confirm client can renew aigaia.com.cy directly through nic.cy

5.4 Week 6-8: DNS Control Recovery

  1. Create new Cloudflare account under client control
  2. Prepare DNS zone file with all current records (ready for quick transfer)
  3. For aigaia.com.cy: Change nameservers in nic.cy to point to new client-controlled Cloudflare
  4. Verify all services continue working after DNS transfer
  5. Fix DKIM records for aigaia.com.cy to restore newsletter deliverability
  6. Fix DKIM records for theatroaigaia.com in existing Cloudflare (or request from provider)

6. New Provider Selection

6.1 Required Capabilities

The new provider must demonstrate competence in:

  1. Domain registration and management (preferably supporting .cy domains)
  2. DNS management with proper SPF, DKIM, DMARC configuration
  3. WordPress hosting with automated backups
  4. E-commerce platform hosting (or willingness to migrate to standard platform)
  5. PHP application hosting if maintaining custom e-shop
  6. SSL certificate management
  7. Office 365 integration experience (not required for hosting, but for DNS configuration)

6.2 Contract Requirements

The contract with any new provider MUST explicitly state:
  • All domains registered in client's name with client as registrant contact
  • Client receives direct access to domain registrar accounts
  • Client receives direct access to DNS management (or shared access)
  • All custom code developed is owned by client (work-for-hire)
  • Regular automated backups accessible to client
  • Clear termination terms including asset handover procedures
  • Notice period requirements (minimum 30 days recommended)
  • Data deletion confirmation upon termination

7.1 If Source Code is Provided

  • Reconfigure BoxNow shipping integration
  • Test thoroughly before DNS cutover

9. Communication Strategy

9.1 Internal Communication

Before announcing to SKWebline:

  • Brief all staff who have contact with SKWebline
  • Ensure no one discusses migration plans until official announcement
  • Prepare staff for potential service disruptions

9.2 SKWebline Communication

Recommended approach: Professional, factual, and documented.

Initial Termination Notice (August 2026)

Send via email with read receipt AND registered post. Include:

  • Formal notice of contract termination effective end of September 2026
  • Request for all assets: domain transfers, source code, database exports
  • Request for Cloudflare account access or DNS zone file export
  • Timeline for handover activities
  • Request for confirmation of data deletion post-termination

Key Principles

  • Keep all communication in writing
  • Remain professional regardless of response
  • Document any refusals or delays
  • Do not make threats or accusations
  • Follow up in writing if verbal agreements are made

10. Legal Considerations

10.1 Contract Review

If a contract with SKWebline is located, review for:

  • Intellectual property ownership clauses (who owns custom code?)
  • Asset ownership clauses (domains, content, data)
  • Termination terms and notice periods
  • Data handling and deletion requirements
  • Dispute resolution mechanisms

10.2 GDPR Considerations

Under GDPR, the client (as data controller) has specific rights:
  • Customer data belongs to the client, not the provider
  • Provider must provide data export in usable format
  • Provider must delete data upon termination (unless legal retention required)
  • Provider cannot use customer data for own purposes

10.3 ICANN Domain Dispute Process

If SKWebline refuses to transfer .com domains:

  • File complaint with ICANN (icann.org/wicf) for registrar non-compliance
  • Contact NETIM directly with proof of ownership
  • Consider UDRP (Uniform Domain-Name Dispute-Resolution Policy) if ownership disputed

10.4 When to Engage Legal Counsel

Consider engaging a lawyer if:

  • SKWebline claims ownership of domains registered in their name
  • SKWebline refuses to provide e-shop source code and claims IP ownership
  • SKWebline takes destructive action causing business harm
  • SKWebline demands payment beyond contracted amounts
  • Contract terms are disputed or unclear
